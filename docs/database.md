# Database

## Overview

The API uses SQLAlchemy 2.x async database utilities and Alembic migrations.

Current files:

- `apps/api/app/db/base.py` defines the shared SQLAlchemy declarative base.
- `apps/api/app/db/session.py` defines the async engine, session factory, FastAPI dependency, and connection check.
- `apps/api/alembic.ini` configures Alembic.
- `apps/api/migrations/env.py` loads application settings and SQLAlchemy metadata.
- `apps/api/migrations/versions/20260618_0001_initial_empty.py` is the first empty migration.
- `apps/api/migrations/versions/20260619_0002_core_tables.py` creates the first core tables.
- `apps/api/migrations/versions/20260629_0003_oanda_index_symbols.py` assigns index CFDs to Oanda.
- `apps/api/migrations/versions/20260629_0004_user_settings.py` adds one-to-one user preferences.

## Connection

The database URL is loaded from `DATABASE_URL`.

Local Compose default:

```env
DATABASE_URL=postgresql+asyncpg://trader:trader@postgres:5432/trading_framework
```

Local host default in code:

```txt
postgresql+asyncpg://trader:trader@localhost:5432/trading_framework
```

## Migration Commands

Apply migrations:

```bash
npm run db:upgrade
```

Show current revision:

```bash
npm run db:current
```

Rollback one revision:

```bash
npm run db:downgrade
```

Create a new autogenerated migration after adding models:

```bash
cd apps/api
alembic revision --autogenerate -m "describe change"
```

Because this workspace may not have Python tooling installed on the host, prefer Docker-backed commands for applying migrations.

## Core Tables

Day 7 adds these tables:

- `users`
- `symbols`
- `candles`
- `watchlists`
- `watchlist_items`
- `user_settings`

The `users` table stores the single local MVP identity used before authentication exists. `get_mvp_user` inserts `MVP_USER_EMAIL` with PostgreSQL conflict handling, so repeated requests resolve the same row instead of creating duplicates.

The default watchlist uses `MVP_WATCHLIST_NAME`. Repository inserts target `uq_watchlists_user_name` and `uq_watchlist_items_watchlist_symbol`, making lazy creation and item addition idempotent under concurrent requests.

Important constraints and indexes:

- `users.email` is unique and indexed.
- `symbols.exchange + symbols.symbol` is unique.
- `candles.symbol_id + candles.timeframe + candles.timestamp` is unique.
- `ix_candles_symbol_timeframe_timestamp` supports candle lookups by symbol, timeframe, and timestamp.
- `watchlists.user_id + watchlists.name` is unique.
- `watchlist_items.watchlist_id + watchlist_items.symbol_id` is unique.
- `user_settings.user_id` is unique and cascades when its owner is deleted.

The settings row stores the active default symbol, supported timeframe, JSON indicator slug list, theme, and IANA timezone. Repository creation uses a conflict-safe insert so concurrent first reads still produce one row per user.

## Repository Layer

Day 8 adds async repositories under `apps/api/app/repositories`.

Shared behavior:

- `create`
- `get_by_id`
- `list`
- `update`
- `delete`

Current repositories:

- `UserRepository`
- `SymbolRepository`
- `CandleRepository`
- `WatchlistRepository`
- `WatchlistItemRepository`

Repository methods flush changes through the active async SQLAlchemy session, but they do not commit automatically. API services should own transaction boundaries so one request can safely write multiple records together.

## Symbol API

Day 9 adds CRUD endpoints for symbols:

- `GET /symbols`
- `GET /symbols/{symbol_id}`
- `POST /symbols`
- `PATCH /symbols/{symbol_id}`
- `DELETE /symbols/{symbol_id}`

`POST /symbols` and `PATCH /symbols/{symbol_id}` return `409 Conflict` when the request would violate the unique `exchange + symbol` constraint.

Backend tests run with `APP_ENV=test`, which makes SQLAlchemy use `NullPool` to avoid keeping asyncpg connections across pytest event loops.

## Candle Cache

Day 14 connects the market-data service to PostgreSQL and uses the existing `symbols` and `candles` tables.

Candles are upserted by:

- `symbol_id`
- `timeframe`
- `timestamp`

The `uq_candles_symbol_timeframe_timestamp` constraint guarantees deduplication. A repeated provider candle updates OHLCV values instead of creating a second row.
